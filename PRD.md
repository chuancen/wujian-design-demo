# 儿童青少年“五健”体检管理平台界面原型 PRD

## 1. 项目目标

基于招标材料截图，建设一个仅包含界面层的 Web 管理应用原型，用于投标演示、截图留证和功能覆盖证明。原型不要求真实后端、真实接口、真实设备接入或真实身份认证，但每项招标功能都需要在页面、组件、状态或弹窗中有明确可见的表达，确保截图可证明“具备该功能入口与业务设计”。

## 2. 原型范围

### 2.1 本期包含

- Web 管理端：医院、学校、科室、年级、班级、项目管理员等角色使用。
- 医护采集端界面：任务授权、扫码入场、体征采集、设备接入、离线同步、异常复核。
- 家长端界面原型：报告查看、异常提醒、复查建议、健康教育。
- 数据驾驶舱：多角色统计、采集进度、质量监控、风险分布、同步状态。
- 系统配置：多机构、多角色、多租户、权限、审计、备份、接口配置。
- 四套差异化设计方案：同一业务覆盖，不同视觉语言。

### 2.2 本期不包含

- 不建设真实后端服务。
- 不接入真实 HIS、LIS、PACS、短信、H5、设备或统一身份认证。
- 不实现真实数据加密、备份恢复、审计存储，只在界面中体现能力。
- 不处理真实儿童个人隐私数据，演示数据全部使用脱敏样例。

## 3. 招标需求拆解

### 3.1 平台总体架构

招标要点：

- 多端协同门户：医院管理端、医护采集端、家长端、学校/区域管理端。
- 私有化部署：医院内网或指定政务云环境。
- 多机构多角色：医院、学校、年级、班级、科室、用户、角色、权限。
- 数据安全：角色权限、操作日志、审计、敏感数据保护、备份恢复。

界面表达：

- 首页展示“多端入口”和“角色视图切换”。
- 系统配置中展示部署环境、租户、组织、权限和审计页面。
- 顶部或侧边栏提供当前机构、角色、数据范围切换。

### 3.2 儿童健康档案

招标要点：

- 一人一档。
- 档案重复校验、档案合并、错误信息修正。
- 跨学期连续管理，展示历次体检趋势。

界面表达：

- 档案列表：姓名、性别、年龄、学校、班级、证件/学籍编号、最近体检、风险标签。
- 档案详情：基础信息、历次体检、五健指标、异常记录、复查记录、家长通知。
- 合并弹窗：重复档案比对、字段冲突、保留策略、合并确认。

### 3.3 基础信息管理

招标要点：

- 学校、年级、班级、学生名单管理。
- 批量导入导出、模板下载、数据校验、错误提示、批量修改。

界面表达：

- 组织树：区域/医院/学校/年级/班级。
- 学生管理表格：导入、导出、模板下载、批量编辑。
- 导入结果页：成功数、失败数、错误行、错误原因、重新上传。

### 3.4 体检任务管理

招标要点：

- 体检批次创建、任务配置、项目配置、参与科室和医护人员配置。
- 支持不同场景配置体检大纲。
- 班级、学校、年级、项目、量表、状态监控。
- 支持一人一码、条码/二维码打印与绑定。

界面表达：

- 任务列表：任务名称、学校、时间、覆盖人数、进度、异常数、同步状态。
- 任务创建向导：基础信息、体检项目、采集流程、人员排班、通知设置。
- 项目配置页：五健、体格、生长发育、视力、脊柱、口腔、心理量表。
- 二维码/条码页：批量生成、打印预览、绑定状态。

### 3.5 医护采集端

招标要点：

- 离线采集、断网暂存、联网同步、冲突校验、异常提示。
- 支持平板、笔记本、台式机等设备。
- 医护人员快速授权：扫码、动态码、账号或院内统一认证。
- 录入校验：身高、体重、视力、腰围等合理范围提醒。
- 常用模块快速选择、漏项提示、异常提示。
- 采集设备对接能力展示。
- 纸质表单补登。
- 数据完整性校验、复核、异常名单生成。
- 随访工作台和家长通知。

界面表达：

- 采集工作台：待检、采集中、已完成、异常、待同步。
- 儿童身份核验：扫码输入、二维码识别、姓名/班级检索。
- 单人采集页：基础体征、视力、口腔、脊柱、心理、营养等分区录入。
- 离线状态条：本地暂存数量、最近同步时间、同步按钮、冲突提示。
- 设备面板：身高体重仪、视力表、血压计、肺活量设备等模拟连接状态。
- 异常复核弹窗：异常字段、规则说明、复核意见、提交人。
- 纸质补登页：按学生/班级/项目快速补录。
- 随访工作台：异常儿童名单、联系状态、随访记录、通知渠道。

### 3.6 五健专项管理

招标要点：

- 营养管理：0-6 岁营养评估、体格检查、生长发育、身高体重 BMI 腰围趋势。
- 视力管理：裸眼视力、矫正视力、屈光筛查、4岁/5岁/6岁筛查节点、中小学生频次。
- 脊柱管理：脊柱/体态采集、复查结果录入、异常程度管理。
- 口腔管理：口腔健康检查、龋齿、牙列、牙周、干预提示。
- 心理行为管理：发育监测、孤独症筛查、诊断访谈记录、心理风险分层。
- 系统化评价：按年龄、性别、学校、项目自动匹配评价标准，形成建议。

界面表达：

- 五健总览页：营养、视力、脊柱、口腔、心理五个专题卡片。
- 专题详情页：指标趋势、异常分布、筛查记录、干预建议、复查计划。
- 评价规则页：年龄段、性别、项目、阈值、等级、建议文案。
- 风险分层页：低/中/高风险人群、筛查依据、处置建议。

### 3.7 家长端

招标要点：

- 家长通过移动端完成基础信息核对、联系方式维护、既往史、过敏史、知情同意填写。
- 查看本次体检报告、历次报告、关键指标趋势。
- 异常提示、复查建议、健康教育资源。

界面表达：

- H5 首页：孩子卡片、当前任务、待确认事项。
- 信息确认页：基本信息、联系方式、既往史、过敏史、知情同意。
- 报告页：五健结果、异常项、趋势图、医生建议、复查入口。
- 健康教育页：按异常类型推荐内容，支持收藏/阅读状态。

### 3.8 院内系统集成

招标要点：

- HIS、LIS、PACS、集成平台、统一身份认证、短信平台对接。
- 检验项目名单导出、结果导入、数据匹配。
- 转诊挂号、预约挂号、线上服务入口。
- 与儿童保健、眼保健、口腔、脊柱、心理、运动康复等专科系统对接。
- 标准接口与二次扩展能力。

界面表达：

- 接口配置页：系统名称、接口类型、地址、认证方式、启用状态。
- 数据交换页：导入任务、导出任务、匹配结果、失败记录。
- 转诊协同页：异常儿童、建议科室、预约状态、回传结果。
- 接口日志页：调用时间、请求方向、状态、错误原因。

### 3.9 可视化统计分析

招标要点：

- 体检进度、异常分布、重点指标趋势、随访与复查完成率。
- 面向不同管理角色展示不同指标。
- 展示健康指标概况、核心指标、健康分布、学校/班级异常风险、近视率、超重肥胖率、脊柱异常筛查率、龋齿干预、心理行为发育监测。
- 展示采集进度、漏项预警、实时检出率、同步状态、现场异常，辅助现场调度。
- 会议、视察、汇报场景安全展示，避免暴露后台管理权限。

界面表达：

- 区域驾驶舱：整体覆盖人数、完成率、异常率、复查率。
- 学校/班级分析：班级排行、风险分布、重点指标趋势。
- 现场调度屏：采集点位、排队人数、漏项预警、设备在线、同步状态。
- 汇报模式：只读大屏，无后台入口，脱敏展示。

## 5. 模块设计要求

### 5.1 导航结构

Web 管理端建议一级导航：

- 工作台
- 体检任务
- 采集管理
- 五健专项
- 儿童档案
- 基础信息
- 随访复查
- 统计分析
- 系统集成
- 系统设置

不同角色看到的菜单应不同：

- 医院管理员：全部菜单。
- 科室医生：采集管理、异常复核、随访复查、儿童档案。
- 学校管理员：基础信息、任务进度、学生档案、家长通知。
- 区域监管：统计分析、汇报模式、学校/班级风险分布。
- 现场负责人：现场调度、漏项预警、设备状态、同步中心。

### 5.2 数据表现规则

- 原型数据必须脱敏，儿童姓名使用样例，如“陈小雨”“李明轩”。
- 每个列表页面至少展示 8-12 行样例数据，方便截图体现真实系统密度。
- 所有核心列表必须有筛选区、状态标签、操作按钮。
- 异常、待复核、待同步、失败、冲突等状态必须用清晰颜色区分。
- 图表不要求真实计算，但需要体现合理指标和趋势。

### 5.3 投标截图策略

每类功能至少准备一张“能自证”的截图：

- 多端多角色：登录/入口页 + 权限矩阵页。
- 档案能力：档案详情 + 档案合并弹窗。
- 基础信息：学生管理 + 导入校验结果。
- 任务能力：任务创建向导 + 二维码打印。
- 采集能力：单人采集 + 离线同步 + 设备状态。
- 五健能力：五健总览 + 专题详情。
- 家长能力：家长报告 + 信息确认。
- 集成能力：接口配置 + 数据交换日志。
- 统计能力：区域驾驶舱 + 现场调度屏 + 汇报只读模式。
- 安全能力：审计日志 + 备份恢复 + 权限配置。

## 6. 四套完全差异化设计方案

四套方案都覆盖同一套页面和功能，但不能做成“同一个系统换皮肤”。每套方案必须像四个不同公司独立完成的产品：信息架构、导航方式、页面密度、组件形态、图表语言、交互路径、文案气质都要明显不同。

统一要求：

- 不共用同一套导航结构。最多保留相同业务名，布局方式必须变化。
- 不共用同一套卡片、表格、按钮、弹窗样式。
- 不只换颜色。每套必须有独立的信息组织逻辑。
- 不让四套页面看起来像同一 UI kit 的不同主题。
- 同一个业务页面在四套方案中截图时，应能一眼看出来自四个不同产品。

### 方案 A：省级政务医疗平台

产品定位：

- 面向医院信息科、卫健委、评审专家。
- 稳重、规范、可信，强调“系统完整性”和“符合政务医疗采购审美”。

信息架构：

- 传统后台结构：左侧多级菜单、顶部机构与角色切换、主区域高密度管理页面。
- 强调“功能完整”和“层级清楚”，菜单直接覆盖招标功能点。
- 页面以表格、查询条件、流程表单、权限矩阵为核心。

视觉特征：

- 主色：医疗蓝、深蓝、白、浅灰。
- 布局：左侧菜单 + 顶部机构/角色切换 + 右侧内容区。
- 组件：高密度表格、状态标签、流程步骤条、标准弹窗。
- 图表：折线、柱状、环图，色彩克制。
- 字体和间距：偏政务系统，字号中等，信息密度高。

交互特征：

- 多条件筛选、批量操作、导入导出、审核确认、标准弹窗。
- 表格行操作清晰，适合证明管理能力和审计能力。

适合截图：

- 权限矩阵、任务列表、档案详情、接口配置、审计日志。

差异重点：

- 最像成熟医院信息系统，适合证明“可上线、可管理、可审计”。

### 方案 B：实时指挥中心产品

产品定位：

- 面向会议汇报、现场视察、领导看板。
- 强调实时进度、风险预警、现场调度和安全只读展示。

信息架构：

- 全屏驾驶舱结构，不使用传统左侧后台菜单。
- 以“实时态势”为中心组织内容：今日任务、采集点位、异常预警、同步状态、复查进展。
- 页面不是管理表格，而是指标墙、队列、时间线、点位监控和告警流。

视觉特征：

- 主色：深黑、墨蓝、青绿、告警橙红。
- 布局：全屏大屏，无传统后台菜单，模块化仪表盘。
- 组件：大数字指标、地图/点位、滚动队列、告警条、进度环。
- 图表：大屏化图表、实时队列、热力分布。
- 字体和间距：大字号、大数字、强对比，适合会议屏幕远距离阅读。

交互特征：

- 以只读展示和场景切换为主：区域总览、学校下钻、现场调度、汇报模式。
- 后台管理入口隐藏或不可见，强调安全展示。

适合截图：

- 区域统计驾驶舱、现场调度大屏、汇报只读模式、异常预警。

差异重点：

- 与方案 A 完全不同，像一家专做数据大屏和指挥调度的公司产物。

### 方案 C：儿童健康服务品牌产品

产品定位：

- 面向家长端、学校端、儿童健康服务场景。
- 视觉更亲和，降低医疗系统的冷感，但仍保持专业。

信息架构：

- 移动端优先，围绕“一个孩子的一次健康服务旅程”组织内容。
- 入口不是系统菜单，而是孩子卡片、待办事项、报告、建议、复查。
- 管理端页面也更像服务运营后台，突出学校、班级、家长触达和健康教育。

视觉特征：

- 主色：薄荷绿、珊瑚橙、天空蓝、暖白。
- 布局：卡片式信息分组，移动端优先。
- 组件：儿童档案卡、健康报告卡、五健图标、建议卡片。
- 图表：趋势曲线、成长曲线、风险等级徽章。
- 字体和间距：更轻、更亲和，图标和插画感更强，但不幼稚。

交互特征：

- 分步确认、轻量问卷、报告解读、健康建议、复查提醒。
- 适合 H5 截图和学校/家长服务场景，不以复杂后台表格为主。

适合截图：

- 家长端首页、家长报告、健康教育、儿童档案详情、五健总览。

差异重点：

- 像一家儿童健康服务公司设计的产品，明显区别于医院信息系统和指挥大屏。

### 方案 D：临床采集效率工具

产品定位：

- 面向现场医护、体检科室、数据录入人员。
- 强调高效录入、低误操作、键盘友好、离线容错和异常复核。

信息架构：

- 不以模块菜单为中心，而以“任务队列 + 当前儿童 + 采集项目 + 异常校验”组织界面。
- 页面常用左右分栏：左侧队列，中间采集表单，右侧设备/规则/历史。
- 大量使用抽屉、命令栏、快捷筛选、状态队列，不做传统行政后台。

视觉特征：

- 主色：白、炭黑、荧光绿或电光紫点缀，不使用政务蓝和儿童彩色。
- 布局：工作台式分栏、固定底部操作栏、右侧实时校验面板。
- 组件：快捷命令按钮、录入单元格、异常校验条、同步队列、设备状态芯片。
- 图表：迷你趋势、项目完成条、字段级风险提示。
- 字体和间距：紧凑、工具化，强调录入效率。

交互特征：

- 快速切换学生、项目间跳转、键盘录入、批量补登、离线同步、冲突处理。
- 弹窗尽量少，更多使用侧边抽屉和内联错误提示。

适合截图：

- 医护采集工作台、单人采集录入、离线同步中心、设备状态、异常复核、纸质补登。

差异重点：

- 像一家专做临床一线工具的公司产物，重点不是漂亮报表，而是高效完成采集任务。

## 8. 原型验收标准

- 每个招标功能点至少能在一个页面上看到明确入口、字段、状态或弹窗。
- 每套设计至少包含 8 个可截图页面，四套合计不少于 32 个截图场景。
- 核心功能截图应能独立说明功能，不依赖口头解释。
- 所有页面只使用脱敏演示数据。
- 不出现“接口未接入”“开发中”“暂无数据”等削弱投标观感的文案。
- 页面文字应贴近招标术语，例如“五健专项”“一人一档”“离线同步”“HIS/LIS/PACS”“复查随访”“权限审计”。

## 10. 后续实施建议

第一阶段先做静态前端原型，保证页面覆盖、截图完整和四套风格差异。第二阶段再按需要补充模拟交互，例如切换角色、打开弹窗、查看详情、模拟同步、模拟异常复核。第三阶段再决定是否接入本地假数据或轻量 mock API。
